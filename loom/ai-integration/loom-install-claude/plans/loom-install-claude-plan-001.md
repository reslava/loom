# CLAUDE-LOCAL.md user-owned rules surface

## Goal

Fix the data-loss footgun where re-running `loom install` overwrites a project's local AI rules. Today `installWorkspace` writes `.loom/CLAUDE.md` unconditionally (it must, to deliver Loom contract updates), so any project that put its own rules there loses them on the next install. The fix establishes clean ownership separation: Loom owns `.loom/CLAUDE.md` (regenerated every install), the user owns a new root `CLAUDE-LOCAL.md` (created once if absent, never overwritten — not even on `--force`), and the root `CLAUDE.md` imports both (`@.loom/CLAUDE.md` then `@CLAUDE-LOCAL.md`). A new rule in the `LOOM_CLAUDE_MD` template tells the AI to keep local rules in `CLAUDE-LOCAL.md` and never hand-edit `.loom/CLAUDE.md`. This is downstream-install-only — it does not touch this repo's recursive root `CLAUDE.md` or the claude-md-sync parity test. Final step extracts a clean `CLAUDE-LOCAL.md` for chord-flow from its saved backup so Rafa can recover the rules already overwritten there.

### Step 1 — Add CLAUDE-LOCAL.md user-owned surface

New `CLAUDE_LOCAL_MD` constant: a short labeled stub (H1 + one-line comment) telling the user this is the home for project-local AI rules and that Loom never overwrites it. New install step writes root `CLAUDE-LOCAL.md` only when `!existsSync` — unconditionally skipped if present, ignoring `--force` (it is user content, outside the Loom-owned set). Extend `InstallWorkspaceResult` with `claudeLocalMdWritten: boolean` and return it.

<!-- step:ensure-both-imports-in-root-claude -->
### Step 2 — Ensure both imports in root CLAUDE.md idempotently

Replace the single-import prepend logic. Compute which of the two import lines are missing from the existing content; if any are missing, prepend the missing ones (Loom contract first, local second) preserving existing body. File-missing → create with both lines. Both already present → no-op (rootClaudeMdPatched=false). Keep the existing `includes()` substring guard semantics per line so re-runs never duplicate.

<!-- step:add-ownership-rule-to-loom-claude -->
### Step 3 — Add ownership rule to LOOM_CLAUDE_MD template

Template-only content — no <!-- rule:{id} --> marker, since the Loom repo itself does not use .loom/CLAUDE.md, so the claude-md-sync parity test must NOT gain a new shared id. Place it near the top (after 'What Loom is') so any AI reading the installed contract sees the ownership boundary early.

<!-- step:update-cli-install-output -->
### Step 4 — Update CLI install output

Add a line for result.claudeLocalMdWritten (written/already exists). Update the rootClaudeMdPatched message text from '(@.loom/CLAUDE.md added)' to reflect both imports are ensured.

Match the existing tests' harness style (ts-node, tests/tsconfig.json).
