### Root cause

`.loom/CLAUDE.md` is a **Loom-owned, regenerated** file. `installWorkspace` (step 2 in `packages/app/src/installWorkspace.ts`) writes it **unconditionally** on every `loom install` — `deps.fs.writeFileSync(loomClaudeMdPath, LOOM_CLAUDE_MD)` with no guard. It *has* to: that's the only way contract updates (like the v1.4.0 `claude-md-sync` work) reach downstream projects. So `.loom/CLAUDE.md` is the single worst place to keep local rules — it's designed to be overwritten.

So the real defect isn't "install overwrote my file" — install did its job. The defect is that **Loom never gave you a place to put local project rules**, so the only file that looked right (`.loom/CLAUDE.md`) was the one Loom owns. We invited the mistake.

### Your proposed fix is the correct shape — I agree with all of it

1. Loom owns `.loom/CLAUDE.md` (regenerated, never hand-edited).
2. The user owns a separate `CLAUDE-LOCAL.md` — Loom creates it **once if absent**, then never touches it again (not even on `--force`).
3. Root `CLAUDE.md` imports both, Loom contract first so local can augment/override:
   ```
   @.loom/CLAUDE.md
   @CLAUDE-LOCAL.md
   ```
4. A contract rule in the `.loom/CLAUDE.md` template telling the AI: *local rules go in `CLAUDE-LOCAL.md`; `.loom/CLAUDE.md` is regenerated by `loom install` and must never be hand-edited.*

This is clean ownership separation — the right architecture, not a merge/patch. Loom regenerates its file freely; your rules are never in the blast radius.

### The install change, concretely

- **New:** create `CLAUDE-LOCAL.md` at repo root *only if absent* (a small labeled stub). Never overwrite — `--force` reinstalls Loom-owned files but must not clobber user content.
- **Root-`CLAUDE.md` patch:** ensure *both* import lines are present at the top, idempotently. Today it only inserts `@.loom/CLAUDE.md` (step 3); it needs to also insert `@CLAUDE-LOCAL.md`, and tolerate the case where one is already there but not the other.

This is downstream-only: it doesn't touch this repo's recursive root `CLAUDE.md` or the `claude-md-sync` parity test. The new template rule is template-only (no `<!-- rule:{id} -->` marker), since the Loom repo itself doesn't use `.loom/CLAUDE.md`.
