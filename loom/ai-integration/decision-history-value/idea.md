# Surface Loom's decision-history value in positioning docs

## What we want to build

Add a clear paragraph (main `README.md`, and likely `docs/WAYS-TO-USE-LOOM.md` / `USER_GUIDE.md`) that names one of Loom's strongest, currently-underplayed value propositions: **a Loom project carries the full, auditable history of every decision.**

## The value to articulate

In any Loom project, owners, collaborators — anyone — can read and trace the **whole decision history**: every idea, design, req, plan, done-note, and the chats that led to them. You never lose the *why*.

Contrast:
- **Without Loom:** you have the codebase, plus whatever docs a disciplined dev happened to write. The reasoning behind decisions is mostly gone — and it's *far* worse when an **AI chat window** was the main vehicle, because that history is ephemeral and unsearchable. Details, trade-offs, and rejected alternatives evaporate.
- **With Loom:** the decision trail is a first-class, durable, greppable artifact of the repo itself.

## Why it matters

- This is a **differentiated** value, not a generic one — it's a direct consequence of "markdown docs are the database" and maps straight to the vision's *traceable / durable memory* pillar. It deserves to be stated plainly, not left implicit.
- It's a strong argument for teams and for long-lived projects (onboarding, handoff, "why did we do this?" six months later).

## Success criteria

- A concise, compelling paragraph on decision-history/traceability lands in the main README (and cross-referenced where it fits).
- It's framed as a concrete benefit with the with/without contrast, not marketing fluff.
